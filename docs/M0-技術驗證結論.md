## 2. M0 期間做出的技術決策

| 決策 | 選擇 | 理由 |
|---|---|---|
| 前端形態 | SvelteKit 2 + Svelte 5 + adapter-static（SPA） | Tauri 官方維護模板；static adapter 編譯為純前端，執行期輕量 |
| russh 密碼學後端 | **`ring`**（非預設的 `aws-lc-rs`） | `aws-lc-rs` 在 Windows 需 cmake/NASM、易失敗；`ring` 內附預編譯組語，僅需 MSVC。保留 `rsa` 特性以支援常見 RSA 主機/使用者金鑰 |
| 終端輸出 IPC | Tauri `Channel<String>`，內容為 base64 | 保證可運作且高效（~1.33x）。原始位元組 Channel 的吞吐最佳化列為後續測量項 |
| Session 模型 | actor-per-session（每連線一 tokio 任務 + mpsc 指令通道） | 單一 session 失敗不影響其他（§6.4）；連線 `Handle` 以 `Arc` 共用，供 SFTP 等額外 channel 使用 |
| known_hosts | app 私有檔（OpenSSH 格式，置於 app config dir） | 與系統 SSH 互通格式，但不污染使用者 `~/.ssh`（Q4 決策） |
| 錯誤模型 | `thiserror` 分類錯誤 + 序列化為 `{kind, message}` | 對應 §6.4 清楚分類的錯誤訊息 |

## 3. 解析出的關鍵 API（russh 0.61.2，供 M1 參考）

- 認證：`Handle::authenticate_password(user, pass).await?.success()`（公鑰 `authenticate_publickey` + `PrivateKeyWithHashAlg`、keyboard-interactive `authenticate_keyboard_interactive_start/respond` 留待 M1 TM-2）
- 通道：`channel_open_session` → `request_pty` → `request_shell`；輸入 `data_bytes`、resize `window_change`、讀取 `channel.wait() -> ChannelMsg::Data`
- 主機金鑰：`russh::keys::check_known_hosts_path`（`Ok(true)` 命中 / `Ok(false)` 未知 / `Err(KeyChanged)` 變更）、`learn_known_hosts_path`、`pubkey.fingerprint(Sha256)`
- SFTP：`channel.request_subsystem(true, "sftp")` → `SftpSession::new(channel.into_stream())`；`read_dir`/`open_with_flags`/`File` 實作 tokio `AsyncRead/Write`
- russh 0.61 連帶引入現代密碼學鏈（curve25519、aes-gcm、chacha20-poly1305、ed25519、ml-kem 後量子 KEM、zeroize），契合 §6.1 現代演算法要求

## 4. 已知限制與延後項（移交 M1/M2）

- **實機端對端測試（後端）**：✅ 已用 Docker OpenSSH（`lscr.io/linuxserver/openssh-server`）+ Rust 整合測試 `ssh::tests::m0_shell_sftp_and_hostkey` 驗證：密碼認證 + PTY shell、shell 串流（送 `echo` 讀回）、SFTP 列目錄/上傳/下載往返、主機金鑰 TOFU 與重連比對皆通過。**GUI 端**（Tauri Channel base64 + xterm 渲染）仍可選擇以 `tauri dev` 連線目視確認（風險低）。
- **認證方式**：M0 僅密碼；公鑰（含 passphrase）與 keyboard-interactive 於 M1（TM-2、AK-2）。
- **主機金鑰 UX**：目前未知主機採 TOFU 自動記錄；M1 需加入「首次連線顯示指紋並由使用者確認」「金鑰變更明顯警示」的互動 UI（TM-6、M1-F8）。
- **輸出 IPC 最佳化**：base64 String channel 可用；M1/效能階段評估原始位元組 channel 與背壓，量測 §6.2（<50ms、cat 大檔不凍 UI）。
- **SFTP 連線復用**：M0 每次操作開新 SFTP channel；M2 改為每連線常駐單一 `SftpSession`，並加上傳輸佇列/進度/續傳（FT-4、FT-7、FT-9）。
- **Session 生命週期事件**：M0 未向前端發送「連線關閉 / 斷線」事件；M1-B5 補上狀態事件以更新分頁狀態。
- **多分頁**：M0 為單一 session UI；多分頁（TM-7）於 M1。

## 5. 驗證指令

```bash
npm run check     # 前端型別檢查（0 errors）
npm run build     # 前端 production build（SPA 產出於 build/）
cargo build --manifest-path src-tauri/Cargo.toml   # 後端編譯（0 warnings）
npm run tauri dev # 啟動桌面 App（需 SSH 目標方能完整體驗）

# 實機驗證（需 Docker）：
docker run -d --name ammax-sshd -e PASSWORD_ACCESS=true -e USER_NAME=amos -e USER_PASSWORD=ammax123 -p 2222:2222 lscr.io/linuxserver/openssh-server:latest
cargo test --manifest-path src-tauri/Cargo.toml --lib ssh::tests -- --ignored --nocapture
```

## 6. 結論

russh 0.61（`ring` 後端）在 Windows 上順利編譯，client API 足以涵蓋 PTY 互動 shell、視窗同步、主機金鑰驗證與 SFTP；與 Tauri `Channel` + xterm.js 的串流架構成立。**無需改用備案 `ssh2` crate**。技術風險已大幅降低，建議依里程碑進入 **M1（MVP）**：站台 CRUD、安全憑證儲存（keyring）、完整認證、多分頁與 SFTP 基本操作，並補齊 M0 延後的 UX 與測試。
