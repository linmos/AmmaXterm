## 1. 產品概述

### 1.1 背景與動機
MobaXterm 是 Windows 上常見的整合式遠端工具，但功能龐雜、僅限 Windows、且免費版有 session 數量限制。許多開發者與維運人員實際上只用到其中幾項核心能力：**SSH 終端機、SFTP 檔案傳輸、站台管理**，以及進內網常需要的**連接埠轉發**。AmmaXterm 的目標是把這些事做到簡單、快速、安全，並以**開源、跨平台、輕量（Tauri + Svelte）** 的形式提供。

### 1.2 產品願景
讓使用者用一個畫面就能管理所有遠端主機、開啟穩定的終端機、在本機與遠端之間順暢傳檔、並按需建立安全通道——啟動快、操作直覺、預設安全、原始碼公開可稽核。

### 1.3 產品目標
- **聚焦**：SSH 終端、SFTP、站台管理、連接埠轉發四件事做到位，不做其餘。
- **效率**：終端輸入延遲低、傳檔吞吐接近線速、開站台只需數秒。
- **安全**：憑證不落地明文、強制主機金鑰驗證、預設使用作業系統金鑰庫、轉發預設綁 localhost。
- **輕量開放**：Tauri 系統 webview、跨平台、MIT 授權、易於社群貢獻與自行建置。

## 2. 範圍

### 2.1 本期範圍（In-Scope）
1. SSH 2.0 終端機（密碼、金鑰、keyboard-interactive 認證），含**多分頁**
2. SFTP 檔案瀏覽與傳輸
3. 站台 / 連線管理（含分組、搜尋、匯入匯出）

5. 金鑰與憑證管理（產生、匯入、安全儲存）

### 2.2 非本期範圍（Out-of-Scope）

- 內建命令列工具集（MobaXterm 的 Cygwin 那一套）
- **團隊協作 / 站台雲端同步 / SSO**：依 D2 先做單機，保留為未來方向（資料模型預留可同步彈性，本期不實作後端與帳號系統）

## 3. 目標使用者與情境

### 3.1 使用者輪廓（Persona）
- **後端 / 全端工程師**：日常 SSH 進開發機與測試機，偶爾傳設定檔、看 log、部署。
- **維運 / SRE**：管理數十至上百台主機，重視分組、跳板機、批次連線、通道與穩定性。
- **資料 / 研究人員**：連 GPU 機或運算節點，常需上傳資料集、下載結果，重視大檔傳輸與斷點續傳。

### 3.2 關鍵使用情境
1. **第一次連線**：新使用者輸入 host / user / 密碼，三步內看到 shell，並被詢問是否儲存為站台。
2. **每日快速進站**：打開軟體，從清單或搜尋框找到站台，雙擊即連，自動套用金鑰。
3. **多工**：同一畫面開多個分頁，分別連不同主機或同一主機多個 session。
4. **看 log 同時傳檔**：在終端機 `cd` 到某目錄後，SFTP 面板自動跟隨到同一路徑，直接把檔案拉進去。
5. **跨跳板機進內網**：站台設定一台 bastion，連線時自動先過跳板再到目標主機。
6. **透過通道存取內網服務**：對某站台設定 local forwarding，把遠端資料庫的 5432 映射到本機，本機工具即可直連。
7. **斷線復原**：網路中斷後自動重連，並還原 session（盡力而為，不保證 shell 狀態）。

## 6. 非功能需求

### 6.1 安全（最高優先）
- 所有憑證預設存於 OS 金鑰庫（透過 `keyring` crate）；本機憑證庫若存在，須以強加密（如 AES-256）保護。
- 強制主機金鑰驗證；金鑰指紋變更須明顯警示並要求使用者確認。
- 支援現代演算法（如 Ed25519、curve25519-sha256、AES-GCM、ChaCha20-Poly1305），並可選擇性開啟相容舊裝置的演算法。
- **連接埠轉發安全**：本機監聽預設綁 `127.0.0.1`；Remote forwarding 預設關閉，啟用時警示暴露風險（見 PF-7）。
- **遙測**：AmmaXterm 為開源，**預設不蒐集任何遙測資料**；若未來加入，須為 opt-in 且僅限匿名使用統計。原始碼公開本身有助於安全稽核。
- 提供漏洞通報流程（`SECURITY.md`）；記憶體中的密碼 / 私鑰使用後盡速清除。

### 6.2 效能
- 終端機本機處理延遲目標 < 50 ms；高吞吐輸出（如 `cat` 大檔、`top`）不得造成 UI 凍結（xterm.js + WebGL addon）。
- SFTP 單檔傳輸吞吐在區域網路下應接近磁碟 / 網路上限；大檔（>1 GB）穩定不中斷。
- 應用程式冷啟動目標 < 2 秒；同時開啟 10+ 個 session / 數條通道時，記憶體成長線性可控。Tauri 系統 webview 有助於壓低基礎記憶體。

### 6.3 相容性
- 終端機相容主流 shell 與 TUI 程式（bash / zsh / fish、vim、tmux、htop 等）。
- 連線端相容主流 SSH 伺服器（OpenSSH 為主），涵蓋常見 KEX / Cipher / MAC。
- 應用程式須在 Windows / macOS / Linux 維持一致的核心體驗（D1）；需特別測試 Linux 端 WebKitGTK 行為差異。

### 6.4 可用性與可靠性
- 連線失敗須有清楚錯誤訊息（DNS / 逾時 / 認證失敗 / 金鑰不符 / 通道錯誤分別提示）。
- 重要破壞性操作（刪除遠端檔案、刪除站台）須二次確認。
- 例外不得導致整個應用崩潰；單一 session 或單條通道失敗不影響其他（每個 session / 通道為獨立 async 任務）。

### 6.5 在地化與無障礙
- 介面字串外部化，至少支援繁中與英文。
- 支援鍵盤操作主要流程；配色具足夠對比。

## 7. UI / UX 概念

### 7.1 整體版面
建議採三區式單視窗：
- **左側站台側欄**：分組樹狀清單 + 頂部搜尋框 + 「快速連線」。可收合。
- **中央工作區**：分頁式終端機（每個 session 一個 tab，可拖曳排序）。
- **下方 / 側邊面板**：可切換顯示 **SFTP 檔案** 與 **通道管理** 兩個分頁，含傳輸佇列與通道狀態。

### 7.2 關鍵畫面
1. **首次啟動 / 空狀態**：引導「新增第一個站台」或「快速連線」。
2. **站台編輯對話框**：基本頁（host/port/user/auth）+ 進階頁（金鑰、跳板機、通道、終端機、外觀）。
3. **連線中終端機**：分頁列、連線狀態指示、右上角開關 SFTP / 通道面板按鈕。
4. **SFTP 面板**：路徑列（可手動輸入、可跟隨終端機）、檔案表格（名稱 / 大小 / 權限 / 修改時間）、底部傳輸佇列。
5. **通道管理面板**：通道清單（類型 / 本機位址 / 遠端位址 / 狀態 / 連線數）、新增 / 啟停。
6. **金鑰管理頁**：金鑰清單、產生 / 匯入、複製公鑰。

## 8. 技術架構（D7：Tauri 已定）

### 8.1 技術堆疊
- **桌面框架**：Tauri（v2）。使用系統 webview（Windows: WebView2、macOS: WKWebView、Linux: WebKitGTK），安裝體積與記憶體遠小於 Electron，最貼近「輕量」定位。
- **後端語言**：Rust。
- **前端框架**：**Svelte**（D10）。編譯為精簡 JS、執行期負擔低，契合輕量定位；與 Tauri webview 整合單純。
- **終端機渲染**：**xterm.js**（搭配 WebGL/Canvas addon），以一般相依整合進 Svelte 前端。

### 8.2 建議函式庫

| 用途 | 建議 | 備註 |
|---|---|---|
| SSH 連線 / channel | `russh`（async, tokio） | 支援 channel forwarding，可實作第 5.4 節通道 |
| SFTP | `russh-sftp` | 與 russh 搭配 |
| 憑證儲存 | `keyring` crate | 跨平台包裝 Credential Manager / Keychain / Secret Service |
| 終端機渲染 | xterm.js + WebGL addon | 應付高吞吐輸出 |

> 授權以實際版本 LICENSE 為準：`russh` 一般為 Apache-2.0，可相容於本專案的 MIT（須遵守各相依授權並附第三方授權聲明）。若偏好成熟的 C 綁定，可改用 `ssh2` crate（libssh2，BSD 類），但為同步式，與 Tauri 非同步事件流整合較不順，故以 `russh` 為主要建議。

### 8.3 架構要點
- SSH 連線、SFTP、通道皆在 Rust 後端以 async 任務處理。
- 終端機位元組串流透過 Tauri 事件 / channel 推送到前端，餵給 xterm.js；輸入、resize 由前端送回後端。
- 每個 session / 通道為獨立 async 任務，單一失敗不影響其他（對應 6.4）。
- 主機金鑰於 russh 的驗證 handler 內檢查，存成 `known_hosts` 相容格式，便於與系統 SSH 互通。
- 站台設定以本機檔案儲存（憑證部分交 OS 金鑰庫），資料結構預留未來可同步的彈性（A4）。

## 12. 風險與緩解

| 風險 | 影響 | 緩解 |
|---|---|---|
| Tauri webview 跨平台一致性（尤其 Linux WebKitGTK） | 體驗在不同平台不一致 | 三平台 CI 測試、避免依賴非標準 webview 行為、建立平台相容性清單 |
| Rust + Tauri 貢獻者池較 JS 小（D7 已知取捨） | 社群成長較慢 | 完整文件、清楚模組邊界、簡單的從原始碼建置流程 |
| 終端機相容性問題（TUI / 控制序列） | 使用者體感「怪」 | 採成熟模擬器（xterm.js），建立相容性測試清單（vim/tmux/htop 等） |
| 大檔 / 大目錄造成卡頓 | 效能口碑受損 | 背景 async 傳輸 + 列表虛擬化 + 分塊並行傳輸 |
| 連接埠轉發遭濫用 / 內網暴露 | 安全事故、信任受損 | 預設綁 `127.0.0.1`、Remote forwarding 預設關閉並警示、清楚顯示使用中通道（PF-7） |
| 開源相依套件授權不相容 | 授權衝突、法務風險 | 建立相依授權清單、CI 加入授權檢查、優先選 MIT/Apache/BSD 相依 |
| 開源維護 / 社群負擔 | 維護者耗竭、回應變慢 | 明確 roadmap 與貢獻流程、自動化 CI/release、設定回應期望 |

## 13. 名詞與備註
- **站台（Site / Session）**：一筆已儲存的遠端連線設定。
- **跳板機 / Bastion（ProxyJump）**：先連到中介主機再連目標主機。
- **連接埠轉發（Port Forwarding）**：Local（`-L`）/ Remote（`-R`）/ Dynamic SOCKS（`-D`）三種通道。
- **edit-in-place**：在本機編輯器開啟遠端檔案、存檔後自動回傳。
- 本文件為 v0.5（定稿），產品命名為 **AmmaXterm**；D1~D10 全數確認，無待確認決策。範圍與技術選型已足以驅動設計與開發。
